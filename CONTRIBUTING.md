# Contributing

Thanks for your interest in contributing to Darkroom Engineering's open-source projects. These are the default guidelines for the [`darkroomengineering`](https://github.com/darkroomengineering) organization. Individual repositories may add their own `CONTRIBUTING.md`, which takes precedence.

## Before You Start

- **Search first.** Check existing [issues](https://github.com/orgs/darkroomengineering/repositories) and pull requests before opening a new one.
- **Open an issue for non-trivial changes.** For anything beyond a small fix, start a discussion in an issue so we can align on the approach before you invest time. This avoids PRs that have to be turned away.
- **Read the repo's README** for project-specific setup, scripts, and conventions.

## Reporting Bugs

Open an issue using the bug report template and include:

- What you expected to happen, and what actually happened.
- A minimal reproduction. A CodeSandbox, StackBlitz, or small repo is ideal.
- Versions (package, framework, browser/OS where relevant).

## Suggesting Features

Open an issue using the feature request template. Describe the problem you're trying to solve, not only the solution you have in mind.

## Pull Requests

1. **Fork** the repository and create a branch from `main`.
2. **Keep PRs focused.** One logical change per pull request.
3. **Match the existing style.** Most repos run linting and formatting, so make sure checks pass locally.
4. **Update docs and types** alongside code where relevant.
5. **Write a clear description.** What changed, why, and how to verify it. Link the related issue.

## Security

Do not open public issues for security vulnerabilities. See [SECURITY.md](./SECURITY.md) for private reporting.

## License

Unless a repository states otherwise, contributions are accepted under that repository's license (typically MIT). By submitting a pull request, you agree your contribution may be distributed under those terms.
